Formal Verification의 세 가지 질문

FPV (Model Checking)
설계 A ⊨ 속성 P ?
"이 설계가 올바른가?"
입력: 설계 + SVA
→ 설계 자체의 버그를 찾음
LEC
설계 A == 설계 B ?
"조합 로직이 같은가?"
입력: 설계 2개 (FF 구조 동일)
→ 합성이 정확한지 검증
SEC
설계 A ≡ 설계 B ?
"시퀀셜 동작이 같은가?"
입력: 설계 2개 (FF 구조 다를 수 있음)
→ 구조 변환이 정확한지 검증

주의: LEC/SEC는 두 설계가 동일하게 틀려도 "동등"이라 판정합니다 — 버그는 FPV로 먼저 잡아야 합니다.

SVA 주요 패턴

1. Request-Grant Protocol

assert property (@(posedge clk) disable iff (!rst_n)
  req |-> ##[1:3] gnt
); // req 후 1~3 사이클 내 gnt 보장

2. AXI VALID Stability

assert property (@(posedge clk) disable iff (!rst_n)
  axi_valid && !axi_ready |=> axi_valid && $stable(axi_data)
); // VALID 동안 data 유지

3. FIFO Overflow Protection

assert property (@(posedge clk) disable iff (!rst_n)
  (count == DEPTH) |-> !push
) else $error("FIFO overflow");

4. Liveness (Eventually)

assert property (@(posedge clk) disable iff (!rst_n)
  req |-> s_eventually gnt
); // req가 들어오면 언젠가는 반드시 gnt

SEC가 필요한 변환 유형

변환설명LECSEC
Retiming FF가 조합 로직을 넘어 이동 불가 가능
Pipeline 변경 Stage 추가/제거 불가 가능
Clock Gating 전력 최적화로 FF 동작 조건 변경 불가 가능
FSM Re-encoding One-hot ↔ Binary 등 불가 가능
RTL 리팩토링 아키텍처 변경, 동일 기능 불가 가능
단순 합성 RTL → Netlist (구조 유지) 가능 가능

핵심 설정 항목

  • Latency Offset — pipeline 깊이 차이만큼 출력 비교 시점 조정
  • Reset Mapping — 두 설계의 초기 상태 일치시키는 시퀀스
  • Port Mapping — I/O 포트명이 다를 때 수동 매핑
  • Abstraction — 넓은 데이터패스 축소 또는 메모리 black-box

주의사항

  • Latency 오설정 — 동일 설계도 fail로 판정
  • 불완전한 Reset — unreachable 상태에서 시작 → false failure
  • 비결정적 동작 — arbiter tie-breaking 등 별도 처리 필요
  • 상태 폭발 — 큰 설계에서는 bounded proof만 가능

MC가 적합한 경우

  • FSM이 deadlock-free인지 증명
  • AXI 프로토콜 handshake 준수 확인
  • Arbiter fairness / starvation-free
  • FIFO overflow/underflow 불가능 증명
  • 인터럽트 우선순위 정확성
  • Security: privilege escalation 불가능

SEC가 적합한 경우

  • 합성 도구 retiming 후 검증
  • 수동 pipeline stage 추가/제거 후
  • Clock gating 삽입 후
  • FSM 인코딩 변경 후
  • RTL 리팩토링 (아키텍처 변경)
  • IP 교체 (동일 기능, 다른 구현)

Best Practices

  • 작게 시작 — 단일 모듈부터 formal 적용
  • Cover 먼저 — assume이 과도하지 않은지 확인
  • SVA 재사용 — sim과 formal에서 동일 SVA 활용
  • Bounded → Unbounded — BMC로 버그 찾고, 이후 full proof
  • 점진적 복잡도 — 간단한 속성부터 점차 확장

Common Pitfalls

  • Over-constraint — assume이 너무 강하면 false positive
  • Vacuous proof — 전제가 항상 false → 증명 무의미
  • 큰 설계 무리 — SoC 전체를 formal로 돌리려는 시도
  • Reset 누락 — disable iff 없이 리셋 중 위반 리포트
  • Latency 오설정 — SEC에서 정상 설계가 fail
